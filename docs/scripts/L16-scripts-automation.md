## העוזר הדיגיטלי בענן

### הבנת הקונספט של אוטומציה מבוססת JavaScript

* **מטאפורת מנהל החנות (The Shop Manager Metaphor):**
  * **ניהול ידני:** כמו מנהל חנות פיזית שעובר מדף-מדף, בודק חוסרים, ומחליף תגי מחיר אחד-אחד עם טוש ומדבקה.
  * **ניהול אוטומטי:** סקריפטים הם כמו מנהל דיגיטלי בלתי נראה. הוא סורק אלפי מוצרים ומעדכן מחירים בשבריר שנייה.
* **מהו סקריפט בפועל? (The API Wrapper):**
  * קוד JavaScript פשוט הרץ ישירות בענן של גוגל (אין צורך להחזיק מחשב דולק).
  * משמש כ״עטיפה״ (Wrapper) נוחה הניגשת ישירות לנתוני החשבון באמצעות ה-Google Ads API.
  * מאפשר לקרוא נתונים ולבצע פעולות (השהיה, הפעלה, שינוי בידים, החלת תגיות) באופן אוטומטי ומתוזמן.
* **מעבר למגבלות האנושיות:**
  * הסרת צוואר הבקבוק האנושי.
  * מניעת טעויות אנוש ופעולה בזמן אמת בקנה מידה עצום (Scale).

## אסטרטגיית ה-80/20 של האוטומציה

### שלושת סקריפטי החובה להגנה וייעול החשבון

* **A. זיהוי אנומליות (Anomaly Detection):**
  * **הצורך:** מניעת אסונות תקציביים או קריסת מעקבים (0 המרות או זינוק של 500% בקליקים בגלל בוטים).
  * **המכניקה:** השוואת הביצועים היומיים לממוצע הסטטיסטי של אותו יום בשבועות קודמים ושליחת התרעה מיידית במייל או ב-Slack.
* **B. ניהול קצב תקציב (Budget Pacing):**
  * **הצורך:** מניעת ״ייבוש״ החשבון באמצע החודש או אי-ניצול של תקציבים מאושרים.
  * **המכניקה:** חישוב קצב הריצה (Run-Rate) מדי יום ועדכון אוטומטי של התקציבים היומיים עד סוף החודש.
* **C. השהיית מילות מפתח מבוזבזות (Pausing Poor-Performers):**
  * **הצורך:** הגנה על שולי הרווח (Margin Protection) וחיתוך ״שומנים״.
  * **המכניקה:** סריקה יומית והשהיית מילים שעברו את סף ההוצאה המוגדר (למשל פי 1.5 מ-tCPA) ללא המרות ב-30 הימים האחרונים.

🎙️ **Narration:**
כדי שלא נלך לאיבוד בעולם התכנות, אסטרטגיית פארטו מתמקדת בשלושה סקריפטים מרכזיים.

## איתור כפילויות למניעת קניבליזציה

### מניעת פיצול מכרזים (Auction Splitting) בעזרת סקריפט בטוח

* **הבעיה בקניבליזציה:**
  * מילים זהות באותו סוג התאמה ביותר מקבוצה אחת מתחרות בינן לבין עצמן.
  * התופעה מפצלת את המכרזים, מקשה על איסוף הדאטה ומקשה על האופטימיזציה.
* **הפתרון — סקריפט במצב יומן בלבד (Log Only):**

```javascript
function main() {
  const keywordIterator = AdsApp.keywords()
    .withCondition("campaign.status = ENABLED")
    .withCondition("ad_group.status = ENABLED")
    .withCondition("status = ENABLED").get();
  const keywordRegistry = {};
  let duplicatesCount = 0;
  Logger.log("--- מתחיל סריקת מילות מפתח כפולות בחשבון ---");
  while (keywordIterator.hasNext()) {
    const keyword = keywordIterator.next();
    const text = keyword.getText().toLowerCase().trim();
    const matchType = keyword.getMatchType();
    const campaignName = keyword.getCampaign().getName();
    const adGroupName = keyword.getAdGroup().getName();
    const uniqueKey = `${text} ||| ${matchType}`;
    if (keywordRegistry[uniqueKey]) {
      duplicatesCount++;
      Logger.log(`⚠️ מילת מפתח כפולה זוהתה: "${keyword.getText()}" (${matchType})`);
      Logger.log(`   - מופע א': קמפיין [${keywordRegistry[uniqueKey].campaign}] -> קבוצת מודעות [${keywordRegistry[uniqueKey].adGroup}]`);
      Logger.log(`   - מופע ב': קמפיין [${campaignName}] -> קבוצת מודעות [${adGroupName}]`);
    } else {
      keywordRegistry[uniqueKey] = { campaign: campaignName, adGroup: adGroupName };
    }
  }
  Logger.log(`--- הסריקה הסתיימה. סך הכל נמצאו ${duplicatesCount} מילים כפולות. ---`);
}
```

🎙️ **Narration:**
אחת הבעיות הנפוצות בחשבונות היא קניבליזציה של מילות מפתח.
זה קורה כשאותה מילה בדיוק נמצאת בכמה קבוצות מודעות.
במצב כזה נוצר פיצול מכרזים.
זה אומר שאתה מתחרה מול עצמך במכרז של גוגל.
זה מייקר עלויות, וּמפזר את הנתונים שלך.
זה מקשה על האלגוריתם לבצע אופטימיזציה נכונה.
כדי לפתור את זה בצורה בטוחה, נשתמש בסקריפט לאיתור כפילויות.
הקוד כאן מותאם למנוע הסקריפטים המודרני של גוגל.
הוא מוגדר לעבודה במצב יומן בלבד.
הוא סורק את המילים בקמפיינים הפעילים, וּמחפש כפילויות.
אם הוא מוצא כפילות, הוא מדפיס אזהרה ביומן.
הוא מראה לך בדיוק איפה המילים נמצאות.
הסקריפט לא עושה שום שינוי בחשבון הפעיל שלך.
זה מאפשר לך לזהות את הבעיות, וּלתקן אותן בראש שקט.

## פתרון תקלות ותחזוקת סקריפטים

### התמודדות עם שגיאות הרשאה ושינויים טכנולוגיים ב-API

* **תרחיש א׳: שגיאות הרשאות גישה (Authorization Failures):**
  * **הבעיה:** סקריפט פעיל נכשל לפתע ומציג `Authentication failed`.
  * **הסיבה:** אסימון הגישה (OAuth Token) מבוטל עקב שינוי סיסמה של המשתמש שאישר את הסקריפט, הסרת המשתמש מהחשבון, או רענון אבטחה תקופתי של גוגל (כל 90–180 יום).
  * **הפתרון:** כניסה לעורך הסקריפט ולחיצה מחדש על כפתור **Authorize** לאימות הרשאות.
* **תרחיש ב׳: שינויי גרסאות ועדכוני API (Deprecations):**
  * **הבעיה:** שגיאות קומפילציה כגון `TypeError` או הודעה ששפת השאילתות AWQL אינה נתמכת עוד.
  * **הסיבה:** גוגל עדכנה את מנוע הסקריפטים לגרסת ES6 מודרנית המבוססת על Google Ads API החדש.
  * **הפתרון:** המרת שאילתות ל-GAQL (לדוגמה: שימוש ב-`campaign.name` במקום `CampaignName`) ועדכון שיטות בחירה (Selectors) לפי התיעוד הרשמי.

## מבדק שליפה: פתרון שגיאות הרשאה

### מהי הסיבה השכיחה ביותר לשגיאת Authentication failed בריצת סקריפט?

סקריפט אוטומטי לניהול תקציב (Budget Pacing) שרץ מדי יום בחשבון ה-B2C של לקוח נכשל לפתע בגלל שגיאת ״Authentication failed״. מהי הסיבה השכיחה ביותר לתקלה זו וכיצד פותרים אותה?

* **א׳.** גוגל אדס חסמה את החשבון בשל חריגת תקציב של מעל 20% מהמתוכנן.
* **ב׳.** ה-API של גוגל עבר שדרוג גרסה והפסיק לתמוך ב-JavaScript באופן מוחלט.
* **ג׳.** המשתמש שביצע את אישור הגישה (Authorization) לסקריפט במקור שינה את סיסמת הגוגל שלו או הוסר מהחשבון, והפתרון הוא כניסה לעורך הסקריפט ולחיצה מחדש על כפתור Authorize.
* **ד׳.** השרת המקומי עליו מותקן הסקריפט כבה ויש להפעיל מחדש את המחשב של מנהל הקמפיין.

  * **זיהוי אנומליות:** מניעת אסונות וקריסת מעקבים.
  * **ניהול קצב תקציב:** בקרה על קצב הריצה החודשי ומניעת חריגות.

  * חובת שימוש במצב **Preview** לבדיקת לוגים לפני הפעלה רשמית.
